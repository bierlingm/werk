# Sibling Sequence, Reality Inheritance, and Parent Snapshots

## The Problem

Sibling tensions — children of the same parent — currently display in creation order (`ORDER BY created_at ASC`). The `children_by_horizon` method in `tree.rs` sorts by `range_start` but is only used in the dynamics engine, not the TUI. Neither approach reflects the structural logic of Fritz's tension chart, where action steps are deliberately sequenced between current reality (bottom) and the desired outcome (top).

Fritz's position on sequence: it is **chosen**, not computed. The creative process is "dimensional, not linear" (*Path of Least Resistance for Artists*), but within a structural tension chart, the practitioner deliberately orders action steps. This ordering is itself an act of creation — it reflects how the practitioner sees the path from current reality to the vision.

## The Tension Chart Model

From the LifeSMyth Creative Tension Worksheet (Fritz-derived):

```
┌────────────────────────┐
│  Vision / Goal         │  ← parent tension's desired state
└────────────────────────┘
  ↑ Tension resolves
  │ to vision
  │
  ├─ action step N       ─┐
  ├─ action step ...      │  ← child tensions (siblings),
  ├─ action step 2        │    deliberately sequenced
  ├─ action step 1       ─┘
  │
┌────────────────────────┐
│  Current Reality       │  ← parent tension's actual state
└────────────────────────┘
```

Children are the bridge between where you are and where you want to be. Their order is not incidental — it represents the practitioner's chosen sequence for resolving the parent tension.

## Design: Three Changes

### 1. Intentional Placement (Explicit Sequence)

**Principle:** When creating a child tension within a parent that already has children, the tool requires the user to place it among its siblings. This reinforces the habit of deliberate choice and makes the resulting order meaningful — you can look at the sequence later and know that you chose it.

**Data model change:**

Add a `position` column to the `tensions` table:

```sql
ALTER TABLE tensions ADD COLUMN position INTEGER;
```

Position is a nullable integer. Null means "no explicit position" (falls back to default ordering). When set, siblings sort by `position ASC`, then by default ordering for ties or unpositioned tensions.

**Default ordering (when no positions are set):**
- Siblings with horizons sort by `range_end` (earliest deadline first)
- Ties broken by precision (narrower first)
- Siblings without horizons sort after those with horizons
- Final tiebreaker: `created_at ASC`

This default reflects urgency — the tension whose temporal window closes soonest appears first. It closely mirrors what a practitioner would choose when thinking structurally: "what needs to resolve first?"

**TUI interaction — creating a child when siblings exist:**

When `CreateChild` is invoked and the parent already has one or more active children:

1. Complete the normal creation flow (name → desire → reality → horizon)
2. Before committing, show the existing siblings as a numbered list and ask: "Place among siblings"
3. The user navigates to a position (between existing items, at top, or at bottom) and confirms
4. The new tension is inserted at that position; existing positions shift

When the parent has no children, skip the placement step — there's nothing to sequence against.

**TUI interaction — reordering:**

Keybindings to move a tension up/down among its siblings:
- `Shift+K` / `Shift+J` — move up / move down
- `Shift+Home` / `Shift+End` (or equivalent) — move to top / bottom

These update `position` values for affected siblings.

### 2. Reality Inheritance

**Principle:** A child tension exists because of a gap in the parent. If the practitioner doesn't enter an explicit reality for the child, the child's reality is the parent's reality at that moment — because that's the context that generated it.

**Current behavior:** If the user presses Esc during the Reality step of creation, `actual` is set to `""`. But `Tension::new` rejects empty `actual` — so the TUI currently uses the tension name as a fallback for both `desired` and `actual` when skipped. This produces tensions where desired == actual, which is structurally meaningless (no gap).

**New behavior:** When creating a child tension and the user skips the Reality step:
- Look up the parent tension
- Set `actual` to `parent.actual` (the parent's current reality text)
- This provides meaningful default context without requiring the user to type

**No data model change needed.** This is purely a TUI-level behavior change during the creation flow.

### 3. Parent State Snapshot

**Principle:** A child tension was generated by a specific gap in its parent — a particular desired/actual pair at a moment in time. When the parent's reality changes, the original context that generated the child is lost. This matters because:

- The child may no longer be relevant (the gap it was addressing closed or shifted)
- The practitioner should be able to see *why* a child was created: "what was the parent's state when I decided this action step was needed?"
- Changes in parent reality imply a question about child validity — not an automatic invalidation, but a prompt for reflection

**Data model change:**

Add snapshot fields to the `tensions` table:

```sql
ALTER TABLE tensions ADD COLUMN parent_desired_snapshot TEXT;
ALTER TABLE tensions ADD COLUMN parent_actual_snapshot TEXT;
```

These are set once at child creation time and never updated. They are a historical record.

**Population:** When creating a child tension with a `parent_id`:
- Look up the parent tension
- Copy `parent.desired` → `parent_desired_snapshot`
- Copy `parent.actual` → `parent_actual_snapshot`

**Surfacing divergence:**

When the parent's `actual` or `desired` changes (via `update_actual` or `update_desired`), children whose snapshots no longer match gain a new structural signal. The dynamics engine can detect:

```
parent_actual_snapshot != parent.actual  →  "parent reality has shifted since this child was created"
parent_desired_snapshot != parent.desired  →  "parent vision has shifted since this child was created"
```

This is not an error state. It is a structural observation: the context that generated this child has changed. The practitioner decides what to do — re-evaluate the child, reorder siblings, release the child, or acknowledge the shift and continue.

**TUI affordance:** In Gaze view on a child whose parent has diverged from its snapshot, show a subtle indicator: "parent reality has changed since creation" or similar. The practitioner can then inspect the snapshot vs. current parent state.

**Future consideration:** When a parent's reality changes, a structural prompt could surface: "N children were created under a different reality. Review?" This connects to the cooperative structural dynamics work — in a multi-participant system, one person updating the parent's reality affects all children held by all participants.

## Ordering Summary

The final sibling ordering logic becomes:

1. **Explicit position** (if set) — `position ASC`
2. **Horizon end date** (if horizon exists) — `range_end ASC` (earliest deadline first)
3. **Horizon precision** (tiebreaker) — narrower first (DateTime < Day < Month < Year)
4. **Creation time** (final fallback) — `created_at ASC`

Explicitly positioned tensions always sort before unpositioned ones within the same parent. This means a practitioner who manually sequences some children but not others will see their chosen order at the top, with unsorted children below in deadline order.

## Implementation Scope

### sd-core changes
- Add `position: Option<i32>` to `Tension` struct
- Add `parent_desired_snapshot: Option<String>` and `parent_actual_snapshot: Option<String>` to `Tension` struct
- Add database migration for three new columns
- Modify `Store::create_tension_with_parent` to accept and store snapshot values
- Add `Store::update_position` method
- Add `Store::reorder_siblings` method (bulk position update)
- Update `children_by_horizon` in `tree.rs` to use the new ordering logic (position → range_end → precision → created_at)
- Add divergence detection to dynamics engine

### werk-tui changes
- Modify creation flow: when parent has children, add placement step after horizon
- Implement reality inheritance: when user skips reality on a child, inherit parent's actual
- Populate parent snapshots during child creation
- Add reorder keybindings (Shift+J/K)
- Add divergence indicator in Gaze view

### store.rs query changes
- `get_children` query becomes: `ORDER BY position ASC NULLS LAST, created_at ASC`
- Horizon-based ordering is applied in-memory after retrieval (as it is now), or via a more complex query joining horizon data

## Design: Tension Chart Visual Layout

The TUI should mirror the structure of Fritz's tension chart: desire at the top, action steps (children) in the middle, current reality at the bottom. This applies to two views: the **Field View** (navigating siblings within a parent) and the **Gaze View** (expanded detail on a single tension).

### Current Layout

**Field View** (when descended into a parent):
```
  ◉ Parent desired state
  ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◉ child tension 1           ▸ ·····◈·
  ◉ child tension 2             ····◈··
  ◉ child tension 3           ▸ ···◈···
```

The parent's desired state appears as a header with a heavy rule separator. Children are listed below. The parent's reality is not visible — you must Gaze on the parent to see it.

**Gaze View** (expanded on a single tension):

```
  ◉ tension desired state       ▸ ····◈·
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  id       01JXYZ...
  desire   the full desired text
  reality  the full actual text
  horizon  2026-03

    ◉ child 1 desired
    ◉ child 2 desired

  gap      ████████░░░░░░░░  moderate
  conflict ...
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
```

Desire and reality appear together near the top as labeled fields. Children appear in a preview block below. This layout treats desire and reality as metadata rather than as the structural poles of the tension chart.

### New Layout

**Field View** (when descended into a parent):
```
  ◉ Parent desired state
  ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◉ child tension 3           ▸ ···◈···
  ◉ child tension 2             ····◈··
  ◉ child tension 1           ▸ ·····◈·

  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  Parent current reality text
```

The parent's reality now appears at the bottom, separated by a light rule. The children sit between desire and reality — visually occupying the structural gap. The practitioner sees the full tension chart: where they want to be (top), what they're doing about it (middle), and where they are (bottom).

Note: children are listed with the action closest to the vision at the top and the action closest to current reality at the bottom. This matches the tension chart worksheet where action steps progress upward from reality toward the goal.

**Gaze View** (expanded on a single tension):
```
  ◉ tension desired state       ▸ ····◈·
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  2026-03                          01JXYZ...

    ◉ child 1 desired
    ◉ child 2 desired

  reality  the full actual text, word-wrapped
           across multiple lines if needed
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
```

The Gaze view is a compact card. The tension line itself already shows the desired state, so repeating it would be redundant. Instead, the card opens with a metadata line: horizon left-aligned, ID right-aligned (dim). Children appear in the middle. Reality sits at the bottom — the structural ground. Dynamics indicators (gap, conflict, neglect, oscillation) are omitted from the Gaze card; they may be replaced by a simpler signal such as alerts or the last mutation timestamp (this is an open question — the current dynamics fields feel noisy for in-line expansion and may be better suited to a dedicated view or the full gaze).

### Why This Matters

The current layout presents desire and reality as adjacent metadata fields. The new layout makes them the **structural poles** of the view — the practitioner literally looks from reality (bottom) through action steps (middle) toward the vision (top). This is not cosmetic; it changes what the tool communicates:

- Seeing reality at the bottom after scrolling through children reminds you where you actually are
- The vertical distance between desire and reality grows as you add more action steps — visually representing the complexity of the path
- The Gaze card is clean: no redundant desire text (already in the tension line), no noisy dynamics fields — just the structural essentials
- The layout mirrors the worksheet, making the tool's structure immediately legible to anyone familiar with Fritz's method

### Implementation

**Field View (`render_field`):**
- After rendering all sibling tension lines, add a light rule separator and the parent's `actual` text (word-wrapped, dimmed) at the bottom
- The parent tension data is already available in `self.parent_tension`

**Gaze View (`build_gaze_lines`):**
- Remove desire text (redundant with the tension line above)
- First line after separator: horizon left-aligned, ID right-aligned (both dim) — a compact metadata header
- Children preview in the middle
- Reality at the bottom, word-wrapped, just above the closing separator
- Remove gap bar, conflict, neglect, oscillation fields (open question: replace with alerts or last mutation, or defer to full gaze / dedicated view)

**Scroll behavior:**
- In the Field View, the parent reality footer should remain anchored at the bottom of the visible area if possible, or scroll naturally with the content if the list is long enough to scroll
- This is a refinement that can be deferred — simply appending the reality at the end of the content works as a first pass

## Open Decisions: Resolved

### 1. Primary Gaze vs Secondary Gaze

The Gaze has two levels: primary (Enter) and secondary/full (Tab). The question is what belongs in each.

**Primary Gaze** — the minimal information you need to act:

```
  ◉ tension desired state       ▸ ····◈·
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  2026-03              created 2026-03-01
                                 01JXYZ...

    ◉ child 1 desired
    ◉ child 2 desired

  last     reality updated 2h ago
  reality  the full actual text, word-wrapped
           across multiple lines if needed
  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
```

Contents:
- Metadata header: horizon (left), creation date (right of horizon or second line), ID (right-aligned, dim)
- Children preview (the action steps)
- Last event: the most recent mutation — what just happened in relation to this tension, stated concisely (e.g. "reality updated 2h ago", "child created yesterday", "horizon set to 2026-03")
- Reality (the structural ground)

The primary gaze answers: *what is this tension, what's happening with it, and where do I stand?*

**Secondary Gaze (Tab)** — the deeper structural report:

Contents (appended below the primary gaze):
- Gap magnitude bar + label
- Conflict detection
- Neglect detection
- Oscillation pattern detection
- Phase information
- Mutation history / activity timeline
- Parent snapshot comparison (if parent state has diverged)
- Annotations/notes

The secondary gaze answers: *what are the structural dynamics at play, and what's the full history?*

This split is clean: primary = orientation and action, secondary = analysis and reflection.

### 2. Placement Interaction

Three options considered:

**Option A: Cursor-between-slots**
Show existing siblings as a list with visible insertion points between them. The user moves a cursor to an insertion slot and confirms. A highlighted line or ghost preview shows where the new tension will land.
- Pro: highly visual, unambiguous
- Con: requires a new UI mode with its own navigation

**Option B: Position-after prompt**
Show existing siblings numbered. Ask "Place after which? (0 for top)." Quick, keyboard-only.
- Pro: fast, minimal UI
- Con: requires the user to count/remember positions, less intuitive

**Option C: Create-then-move**
Always create the new tension at the bottom (closest to reality — the natural starting point for a new action step). Immediately enter reorder mode. The user moves it with Shift+J/K to the desired position and confirms.
- Pro: reuses the reorder mechanism, one less interaction mode to learn, the user sees the tension in context while placing it
- Con: requires learning reorder keys at creation time (but they'll need to learn them anyway)

**Recommendation: Option C.** It's the most economical — no new UI mode, the reorder affordance does double duty, and it naturally teaches the practitioner the reordering keys during creation. The default position (bottom, closest to reality) is also structurally sound: a new action step is something you're doing *now*, from where you *are*.

### 3. Positioned vs Unpositioned Siblings

Two approaches compared:

**Auto-assign all:** Once any sibling in a group gets a manual position, all siblings receive auto-assigned positions based on their current order. Simple mental model, no visual split. But it lies — it assigns positions the user didn't choose, making it look like deliberate sequence when it isn't. Fritz would call this a misrepresentation of current reality.

**Visual distinction:** Positioned siblings appear in their chosen order. Unpositioned siblings appear below them, subtly distinguished — perhaps slightly dimmed, or separated by a faint dotted line. The tool is honest: "you chose this order for these; you haven't chosen for those."

**Decision: Visual distinction.** It's more structurally honest. The separation is itself information — it tells the practitioner "you have unchosen work here." Over time, the natural pull is to position everything, because the unpositioned section is a visible reminder of uncommitted sequence. The dimming should be subtle — these are still active tensions, not second-class. A faint separator line (thinner than the reality rule) between the two groups is likely sufficient.

Layout with mixed positioned/unpositioned:
```
  ◉ Parent desired state
  ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ◉ child 3 (positioned)     ▸ ···◈···    ← closest to vision
  ◉ child 1 (positioned)       ·····◈·    ← closest to reality
  · · · · · · · · · · · · · ·
  ◉ child 4 (unpositioned)      ··◈····    ← unsorted, by deadline
  ◉ child 2 (unpositioned)    ▸ ····◈··

  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  Parent current reality text
```

### 4. Reality Requirement

**Decision: Require reality for all tensions.** Not just children — every tension must have an `actual` state. This is structurally correct: a tension without a stated reality has no gap, and without a gap there is no structural tension. The current fallback (desired == actual when skipped) is worse than requiring input, because it produces a tension that structurally claims "I'm already there" — which is false and misleading.

This means removing the Esc-to-skip behavior on the Reality step entirely. The creation flow becomes: name → desire → reality (required) → horizon (optional).

Reality inheritance for child tensions still applies: if the user is creating a child, the reality input field can be **pre-filled** with the parent's current reality. The user can accept it (Enter) or type over it. This is different from skipping — they're confirming a default rather than leaving a void.

**Implementation note:** `Tension::new` already rejects empty `actual`. The change is in the TUI: remove the Esc escape from the Reality step, and pre-fill with parent reality when creating children.

### 5. Child Ordering Direction

Children form a bridge from reality to vision. The chart reads bottom-to-top: reality at the bottom, vision at the top. The first action step (what to do first, from where you are) should appear closest to reality.

**Visual layout:**
```
  Vision/Desired (top)
  ━━━━━━━━━━━━━━━━━━

  child N  ← last step, closest to vision
  child ...
  child 2
  child 1  ← first step, closest to reality
  · · · · · · · · · ·
  child X  ← unpositioned (by deadline)
  child Y  ← unpositioned (by deadline)

  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
  Current Reality (bottom)
```

Position 1 = first action from reality = appears at the bottom of the positioned group. Position N = final step before vision = appears at the top. The list renders in **reverse position order** (highest position at top, lowest at bottom).

Unpositioned siblings appear between the positioned group and reality. They haven't been placed in the sequence, so they sit in the "unchosen" space closest to the ground.

**When creating a new child (Option C from above):** The new tension appears at the bottom of the positioned group (position = lowest), closest to reality. The user then moves it up with Shift+K if it belongs higher in the sequence. This is natural: "I'm adding a new step — where in the bridge does it go?"

**Reordering mental model:** Shift+K moves a tension *toward the vision* (up). Shift+J moves it *toward reality* (down). This matches both the visual direction and the structural meaning.

### 6. Divergence Indicator

When a parent's state has changed since a child was created (snapshot ≠ current), the child needs a signal. Options explored:

**Option A: Glyph on the tension line.** A small marker (e.g., `△` or `~`) next to the status glyph, visible without gazing. Pro: always visible, draws attention. Con: adds visual noise to every diverged child, which could be many.

**Option B: In the Gaze card metadata line.** Add a note like `⚠ parent shifted` to the primary gaze. Pro: visible when you look at a specific tension. Con: invisible in the sibling list.

**Option C: Subtle color shift on the tension line.** The desired text renders in a slightly different color (e.g., muted amber instead of white). Pro: non-intrusive, peripheral awareness. Con: may be too subtle, colorblind-unfriendly.

**Option D: Only in secondary gaze.** Show the snapshot comparison in the full structural report (Tab). Pro: no visual noise. Con: you have to actively look for it.

**Option E: On the parent, not the children.** When viewing the parent's children list, the parent's reality footer shows a marker: `reality (changed)` or similar. When viewing children, their lines are normal. Pro: the signal is at the structural source (the parent changed, not the children). Con: doesn't tell you which children are affected (all of them are, by definition).

**Recommendation: B + D combined.** Primary gaze shows a brief note when parent has diverged (e.g., a line `parent  reality shifted` in the metadata area, dim). Secondary gaze shows the full comparison: original snapshot alongside current parent state, so the practitioner can evaluate the delta. No glyph on the tension line itself — divergence is context that matters when you're *looking at* a child, not when you're scanning a list.

Additionally, Option E is worth considering as a complement: the parent's reality footer in the Field View could indicate it has changed relative to when children were created. This is cheap to implement and structurally accurate — the shift happened at the parent level.

### 7. Parent Desire vs Reality Snapshot — Different Signals?

These are structurally different events:

**Parent desire changes:** The goal has moved. Every child was an action step toward a destination that is no longer the destination. This is a fundamental reframe. In Fritz's terms, the structural tension itself has been redefined — not just the current-reality pole but the vision pole. All children are potentially invalidated, not because they failed but because success has been redefined.

**Parent reality changes:** The ground has shifted. The children are still aimed at the same vision, but the starting conditions are different. Some children might now be unnecessary (reality moved toward desire — progress). Others might need reordering (a different starting point suggests a different path). The direction is preserved; the distance and terrain have changed.

**Reflection:** These deserve different treatment, but the difference is in *severity of signal*, not in mechanism. Both use the same snapshot comparison. But:

- **Desire divergence** should be a stronger signal — perhaps the `parent  vision shifted` line in primary gaze, and a recommendation in secondary gaze to review all children.
- **Reality divergence** should be a gentler signal — perhaps only in secondary gaze, since reality changing is the *expected* outcome of doing work. Every time you update a parent's reality to reflect progress, all children would show divergence. This is normal, not alarming.

This suggests: maybe `parent_actual_snapshot` divergence should only surface when the *semantic direction* of the change is significant — but that requires judgment the tool can't make. A simpler heuristic: reality divergence surfaces only in secondary gaze. Desire divergence surfaces in primary gaze. The practitioner evaluates.

### 8. Migration

Existing tensions get `position = NULL`. They sort by the default ordering (horizon end → created_at). This is acceptable — the practitioner's existing experience is preserved through creation-time ordering, and they can manually position tensions going forward.

### 9. Root Level and the Senior Organizing Principle

Fritz is clear: a senior organizing principle is required. Without one, competing values produce structural oscillation — success in one area causes difficulty in another, and the system swings back. Multiple root tensions are structurally analogous to an organization without a clear purpose.

But a tool should observe structure, not enforce ideology. Options:

**Option A: Enforce single root.** One root tension; everything else is a child. Clean, Fritz-correct. But potentially constraining — a practitioner might be genuinely exploring what their senior tension *is*, and forcing a single root prematurely could produce a dishonest placeholder.

**Option B: Allow multiple roots, surface the structural observation.** When more than one root tension exists, the tool notes: "multiple root tensions — consider whether these compete." The observation is Fritz-derived; the decision is the practitioner's. This is how the dynamics engine already works — it detects conflict and oscillation without resolving them.

**Option C: Allow multiple roots, encourage hierarchy.** When creating a second root tension, offer: "Would you like to create a parent tension to unite these?" Not required, but prompted.

**Option D: Allow multiple roots silently.** Current behavior. No opinion from the tool.

**Recommendation: Option B, possibly with C as a refinement.** The tool's job is structural observation. Multiple roots are a structural fact worth surfacing, just like conflict or oscillation. The practitioner decides whether to unify them under a senior tension or to accept the multiplicity.

A subtler point: multiple roots might be *temporarily* valid during exploration (you're figuring out what matters most) but *structurally unstable* as a permanent state. The tool can reflect this by surfacing the observation with increasing urgency over time — similar to how neglect detection works. "You've had competing root tensions for 3 weeks" is a different signal than "you created a second root tension today."

Root tensions should still support explicit positioning and reordering — they are siblings (siblings with no parent). The same placement and visual distinction rules apply. If there's only one root (recommended end state), ordering is moot.

This also opens a design question for the Field View at root level: there is no parent desired state to show at the top and no parent reality to show at the bottom. The root level is structurally different — it *is* the senior organizing principle, not a bridge within one. The root-level Field View might benefit from a different visual treatment, but this can be deferred.

## What This Does Not Address

- **Cooperative dynamics** — parent snapshots create the foundation for multi-participant divergence detection, but the cooperative model itself is a separate design
- **Automatic invalidation** — divergence is surfaced, never acted on automatically
